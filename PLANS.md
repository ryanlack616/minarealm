## 14) Core Strategy (Deep Version)

### Positioning

- Minarealm is a trust-first local destination for crystals and spiritual services.
- The brand promise is clear guidance, genuine products, and practical care.
- The site should reduce anxiety for first-time visitors and increase confidence to call, visit, or book.

### Priority Channels (In Order)

- Google Business Profile (intent-ready local traffic).
- Website conversion flow (turn visits into calls/directions/bookings).
- In-store review capture (proof loop that compounds).
- Social support layer (distribution, not primary conversion engine).

### Who We Are Optimizing For

- First-time local shopper (needs trust + clear next step).
- Service-intent visitor (needs booking confidence + prep clarity).
- Returning customer (needs updates and reasons to revisit).

## 15) Assumptions To Validate In First 30 Days

- Assumption A: Most high-intent users will choose Call or Directions over long forms.
- Assumption B: Testimonials + FAQ will reduce hesitation for first-time service bookings.
- Assumption C: QR review prompts at checkout will increase review velocity without discounting.
- Assumption D: Weekly new-arrivals content increases return visits and walk-ins.

Validation rule:

- Keep assumptions only if metrics improve in a 2-3 week window.

## 16) Funnel Model And KPI Math

### Funnel Stages

- Discover: GBP impressions, local search visibility.
- Engage: page views, time on key sections, CTA attention.
- Act: call clicks, directions clicks, booking clicks.
- Convert: completed inquiry/booking, in-store visit, offer redemption.
- Retain: repeat visit, repeat service, subscriber growth.

### Minimum Weekly Target Stack (Starter)

- 100 local site visits/week.
- 20-30 CTA actions/week (calls + directions + booking clicks).
- 8-12 qualified inquiries/week.
- 2-4 incremental service conversions/week.

### Tracking Definitions

- Call click: any click on tel links.
- Directions click: any outbound maps link click.
- Booking intent: click on service CTA or booking anchor.
- Inquiry: direct message, call outcome, form/SMS request with service intent.

## 36) In-House Shopping Cart + Inventory System — Deep Design

### Why Build In-House

Shopify costs $29–$299/month forever. Square has a storefront but no mineral knowledge layer. Neither can do what this system needs at its core: product pages that are also educational reference entries for each stone — Mohs hardness, crystal system, metaphysical properties, chakra associations, care instructions. Ryan's standard stack applies — FastAPI + SQLite + single-file HTML. No monthly SaaS fees.

---

### The Core Domain Problem: Crystal Retail Is Not Normal Retail

Most e-commerce schemas assume interchangeable units — you buy "a red t-shirt size M" and any one of 50 identical units is fine. Crystal retail has three fundamentally different inventory modes:

**Mode 1 — Unique specimens.** This specific 1.2 kg amethyst cluster is one object. When it's sold it's gone. It has its own photos, weight, dimensions, and grade. Quantity is always 0 or 1. If two customers add it to their cart simultaneously, only one can buy it — the other gets a conflict.

**Mode 2 — Weight-sold bulk.** Raw citrine chips sell by the gram or ounce. Customer chooses a quantity (e.g., 50g) and price is calculated. Stock is a running weight total, not a unit count.

**Mode 3 — Standard units.** A selenite wand sold in a pack of 5, or a specific tarot deck — these are interchangeable units. Normal qty tracking applies.

The schema must handle all three modes cleanly, plus services (non-physical, bookable) and classes/events (ticketed, limited seats).

### The Unique Specimen Race Condition — Full Solution

1. Customer A adds the amethyst cluster to cart → system checks `inventory_items.quantity - reservation_qty > 0`.
2. If yes: set `cart_items.reservation_expires_at = now + 15min`, increment `inventory_items.reservation_qty += 1`.
3. Customer B tries to add the same item → same check, now `quantity - reservation_qty = 0` → reject with "Someone else has this in their cart. Check back in a few minutes."
4. Customer A completes checkout → `inventory_items.quantity -= 1`, `reservation_qty -= 1`, item archived.
5. Customer A abandons cart → background sweep (run on any add-to-cart request, or a cron every 5 min) finds `reservation_expires_at < now` → `reservation_qty -= 1`, item released.
6. For the admin panel: show "Reserved" status on unique specimens with timer.

### Weight-Sold Price Calculation

```python
def cart_item_total(item):
    product = get_product(item.product_id)
    if product.pricing_model == "per_gram":
        return round(product.price_per_unit * item.quantity, 2)
    elif product.pricing_model == "per_oz":
        return round(product.price_per_unit * (item.quantity / 28.3495), 2)
    else:
        return round(product.base_price * item.quantity, 2)
```

The cart UI for weight-sold items shows a quantity slider in grams with a live price update.

### Summary: The Three Design Principles

**1. Task-first, not data-first.**

**2. Crystal knowledge is automatic, not manual.**

## 38) Gap Analysis — Where the Design Is Incomplete

Critical problems, missing specs, and things that will break in practice. Ordered by severity.

---

### GAP 1 — CRITICAL: Hosting Architecture Is Wrong

**The problem:** The current minarealm.shop site lives on Porkbun static FTP hosting (see `_deploy.py`). FastAPI + SQLite requires a persistent server process. Porkbun static hosting cannot run it. You cannot FTP a Python process.

**What breaks:** Everything. You cannot deploy the backend to Porkbun. The entire Phase 1 build never goes live if we don't resolve this first.

**Options:**
- **Option A (recommended):** Deploy the backend to Fly.io — same stack as groundequals.com and atomiclithosphere. Free tier handles low traffic easily. `minarealm.shop` stays on Porkbun (fast, cheap). Backend lives at `api.minarealm.shop` or a Fly URL. Admin panel and storefront call the API.
- **Option B:** Move the whole site to Fly.io — backend + static files served by FastAPI. Eliminates Porkbun. Fly free tier has cold-start latency (~2s first hit) which is acceptable.
- **Option C:** VPS (DigitalOcean $6/mo or similar) — more control, slightly more maintenance.

**Decision needed before Phase 1 starts:** Pick Option A or B. Add to budget (Fly.io free tier may be sufficient; $0–$5/mo).

**Integration point:** The main site (Porkbun static) needs a Shop link that goes to the storefront URL. One `<a>` tag change in `index.html`.

---

### GAP 2 — CRITICAL: Michigan Sales Tax Not Addressed

**The problem:** Physical goods sold online to Michigan customers are subject to Michigan 6% sales tax. Services may be exempt. Not mentioned anywhere in §36 schema or §37 UX.

**What breaks:** Stripe charges have no tax line. Orders show wrong totals. Potential tax compliance issue.

**What needs to happen:**
- Add `tax_rate` and `is_taxable` fields to `products` (physical goods = taxable, services = not).
- Calculate tax at checkout: `tax_amount = subtotal_of_taxable_items * 0.06`.
- Show as a line item on the order summary screen and order emails.
- Add `is_taxable INTEGER DEFAULT 1` to `products` schema (services default to 0).
- Note: Cynthia may already be collecting sales tax in Square in-store. Confirm she has a Michigan sales tax account/permit.

---

### GAP 3 — CRITICAL: Photo Handling Not Fully Specified

**The problem:** §37 says "take a photo or upload from camera roll" but doesn't specify what happens to the file.

**Real-world detail:** Cynthia uses an iPhone. iPhone photos are HEIC format, 4–8MB each. A browser `<input type="file">` will accept them but serving raw 6MB HEIC files on product pages is not acceptable (slow, Safari-only support on other devices).

**What needs to happen:**
- Server-side image processing: accept HEIC/JPG/PNG, auto-convert to JPEG, resize to 1200px max dimension, compress to ~200–400KB. Python Pillow handles this.
- Generate two sizes per photo: `full` (1200px) and `thumb` (400px). Thumb used in grid views, full in product pages.
- File naming: `{inventory_item_id}_{n}.jpg` to avoid collisions.
- Storage: local `images/` directory on the Fly instance is fine for Phase 1, but Fly volumes are ephemeral on free tier deploys — need a persistent volume or Cloudflare R2 / Backblaze B2 (both have free tiers).
- **Ephemeral volume risk:** If Fly restarts the app, local image storage without a persistent volume is wiped. This is a data-loss risk for Cynthia's product photos. Must use a persistent volume or external object storage before going live.

---

### GAP 4 — SIGNIFICANT: Square Sync Direction and Conflict Resolution

**The problem:** §36 says "keep Square for in-person POS, optional `_sync_square.py` syncs inventory." But the sync direction and conflict resolution aren't defined.

**The real conflict scenario:** Cynthia sells a rose quartz cluster in-store via Square at 2pm. A customer added it to their online cart at 1:55pm (reservation active). When Square sells it, the inventory goes to 0 in Square but our system still shows it reserved. Who wins?

**What needs to happen:**
- **Direction: Square is the source of truth for in-store sales.** Our system is source of truth for online sales.
- Sync runs: on-demand via admin button ("Sync with Square") OR on a schedule (every 15 min via APScheduler or Fly cron).
- Conflict rule: if `square_qty < local_qty`, trust Square (something sold in-store). Update local qty down. If an online cart has a reservation for that item, release the reservation and notify the customer with a "Sorry, that item just sold in-store" email.
- Square Catalog API v2 needed. Items need a `square_catalog_object_id` field added to `products` for the link.
- For Phase 1 (admin only, no online sales): sync is one-way, Square → local. No conflict possible.

---

### GAP 5 — SIGNIFICANT: Reservation Sweep Mechanism

**The problem:** §36 describes the 15-minute TTL for unique specimen reservations. But "a background job or on-demand check" is too vague. The specific mechanism matters for architecture.

**FastAPI does not have built-in background scheduling.** Options:
- **APScheduler** embedded in the FastAPI process: simple, runs sweep every 5 min. Fine for low traffic. Adds dependency.
- **On-demand check at add-to-cart:** Every time any user adds to cart, sweep expired reservations first. Zero extra infrastructure. Works if traffic is low (which it will be). **Recommended for Phase 1.**
- **Fly cron:** A separate Fly Machine that runs on a schedule. Overkill for Phase 1.

**Decision:** Use on-demand sweep for Phase 1. Every `POST /cart/add` runs `sweep_expired_reservations()` first. Total reservation sweep is ~1ms for a SQLite table with <1000 rows.

---

### GAP 6 — SIGNIFICANT: Multi-Location Inventory Conflict

**The problem:** Schema has `location: fenton|hartland|both|online_only`. But it doesn't handle: Cynthia transferring a piece from Fenton to Hartland, or an online sale depleting which location's stock.

**Real scenarios:**
- Piece is at Fenton. Online customer orders it for pickup at Hartland. Cynthia needs to transfer it. System doesn't track that.
- Item has `location: both`. Customer buys it online. Which shop's stock goes to 0?
- Item moves from Fenton to Hartland. How does Cynthia update that?

**Minimum viable solution:**
- Add a `transfer_log` table (future) OR handle it simply: `location` is an editable field in the admin with a note "if you move it, update this." For Phase 1, trust Cynthia to keep it honest.
- For online orders of a `location: both` item: prompt at checkout "Pick up at: [Fenton] [Hartland]." That choice becomes the fulfillment location, and the system records which shop's stock was drawn.
- Document this limitation clearly in the admin UI: "If you move a piece to a different location, tap Edit and update where it is."

---

### GAP 7 — SIGNIFICANT: Shipping Is Not Fully Specified

**The problem:** `fulfillment_type: shipping` exists in the schema. But we haven't defined: how are shipping costs calculated, how does Cynthia actually ship (what carrier, does she print labels), and crystal-specific packing requirements.

**Real operational questions:**
- Does Cynthia want to offer shipping at all in Phase 1, or is it local pickup only first?
- Crystals are heavy and fragile. Flat-rate USPS boxes are common in the crystal resale space.
- Does she need a shipping label integration (Shippo, EasyPost, or manual USPS.com)?

**Minimum viable solution for Phase 1:**
- Offer only local pickup in Phase 1. No shipping. Add a banner: "Local pickup only — Fenton or Hartland." Defer shipping complexity to Phase 3.
- This eliminates: shipping cost calc, label printing, carrier integration, Michigan origin zip lookup.
- When shipping is added: flat-rate table by weight bracket is simplest. Shippo has a free tier for label generation.

**Decision needed from Cynthia:** Does she want online shipping at all, or is this a local-only shop?

---

### GAP 8 — SIGNIFICANT: Integration with Existing minarealm.shop Homepage

**The problem:** We designed the shop as a separate project at `C:\rje\dev\minarealm-shop\`. But the existing homepage at `minarealm.shop` has no Shop link, no awareness of the cart system, and no connection to it.

**What needs to happen:**
- Add a "Shop Online" nav link to `index.html` pointing to the storefront URL.
- Shop product pages need their own SEO: `<title>`, `<meta description>`, OG tags (with product photo), and `Product` JSON-LD schema with `offers`, `price`, `availability`, and `brand`.
- The homepage highlights bar (once live) should say "Shop online →" if the store is active.
- Decide: is the shop at `minarealm.shop/shop/` (requires FastAPI to serve everything including the homepage) or at `shop.minarealm.shop` (separate subdomain, keeps Porkbun for homepage)?

**Recommendation:** `shop.minarealm.shop` subdomain served from Fly.io. Homepage stays on Porkbun (fast, no cold start). A CNAME DNS record at Porkbun points `shop` to the Fly app URL. Two-minute DNS change.

### GAP 9 — MODERATE: Bulk Import / Initial Inventory Load

**The problem:** The 7-step wizard is ideal for adding one item at a time. Cynthia likely has 100–300 items in stock right now. Walking through 300 wizard flows to set up the initial inventory is not viable.

**Options:**

**Recommended for Phase 1:** Ryan does initial load from a list Cynthia provides. Add a note to the onboarding flow: "For your first inventory upload, Ryan will help you import your existing stock." Add `_seed_inventory.py` to Phase 1 deliverables.

### GAP 13 — MODERATE: Email Notifications — Both Directions

**The problem:** Three notification flows were assumed but never fully specced:
1. Customer receives order confirmation email
2. Cynthia receives "new order" notification email

**What needs to happen:**

### GAP 14 — LOWER: Sold vs Archived Distinction

**The problem:** §37 mentions both "Mark as Sold" and "Move to archive" as separate actions. But Cynthia may not understand the difference, and the difference matters for inventory reporting.

**Clear definitions needed in the UI:**
- **Sold** = this item was purchased by a customer. Inventory goes to 0. Sale is recorded (even if not through the online store — Cynthia can mark in-store sales manually). Used for sales reporting.
- **Archived** = removed from the shop for any other reason (damaged, pulled, gifted, mistake). NOT a sale. Inventory goes to 0 but no sale recorded.
- Admin shows both as "no longer in stock" but reports separate them.
- Tooltip/help text on both buttons to explain the distinction in plain language.

### GAP 15 — LOWER: Crystal Subscription Box — Schema Gap

**The problem:** Crystal subscription boxes are listed in Phase 3 (§35 Sprint C). But the current schema has no mechanism for recurring billing. Stripe Subscriptions (or Stripe Billing) is a completely different product from one-time `PaymentIntent`. If this is a real Phase 3 goal, it needs to be flagged now — it's not a small add-on.

**Flag:** When subscription boxes become a priority, plan a separate sub-project. It requires: subscription products table, Stripe Subscription object (not PaymentIntent), billing cycle management, fulfillment schedule. Estimate 2–3× the effort of the base cart system. Do not start Phase 3 assuming it's a small feature.

### GAP 16 — LOWER: Pricing Update Mid-Cart Warning

**The problem:** `cart_items.price_at_add` protects the customer's price snapshot. But if Cynthia updates a price on a live product, she won't know she's affecting active carts. A customer with the item in cart will still pay the old price, which might be lower.

**Simple solution:** When Cynthia edits a product price in the admin, check if any active cart has that item. If yes: show a warning toast: "3 people have this in their cart at the old price of $X. They'll still be charged that amount if they complete checkout." No action required — just awareness.

## 41) Online Ordering — Real Revenue Build Plan (May 4, 2026)

### Context and decision

Cynthia wants to start making money and shipping. The current shop at `minarealm.shop/shop/` is order-intent capture only — it collects name/email and fires a "pending_invoice" order into the worker KV, then Cynthia manually sends a Square invoice. That pipeline works for pickup but breaks for real online sales: stock decrements before payment, there is no shipping address collection, no tax, no payment confirmation, and no customer receipt.

All forms are already internal (Formspree action attributes are dead code — JS intercepts with preventDefault and POSTs to the Cloudflare Worker). No Formspree dependency exists. Formspree attrs, `_subject`, and `_gotcha` fields removed May 4, 2026.

**Decision: use Square Payment Links API for checkout. Cynthia already has Square — it is linked to her bank, handles tax, sends receipts, and she knows the dashboard. No new payment processor onboarding.**

---

### Part 1 — What "real online ordering" requires

To actually ship product and collect money, every order needs:

1. **Real payment** — charge at time of order, not "invoice to follow". Square Payment Links.
2. **Shipping address** — street, city, state, zip, country. Required for label generation.
3. **Shipping cost** — flat rate tiers by weight bracket (Phase 1) or Shippo API quote (Phase 2).
4. **Michigan sales tax** — 6% on taxable retail goods. Square already handles this if her Square account has Michigan tax configured (check Square Dashboard → Tax). If configured, Square collects and remits — no manual calculation needed.
5. **Order confirmation email** — Square sends customer receipts automatically.
6. **Stock hold vs stock decrement** — stock reserved on Payment Link creation, decremented on payment webhook confirmation.
7. **Cynthia notification** — Square notifies her of new payments. Also add MailChannels worker email for ship-needed orders.
8. **Label workflow** — Cynthia prints USPS/UPS label. Shippo ($0.05/label) Phase 2.

---

### Part 2 — Architecture (Square Payment Links)

Everything stays in the Cloudflare Worker. Square is the only new dependency — Cynthia already has the account.

**Flow:**
1. Customer fills cart on `shop/index.html`, selects shipping or pickup
2. For **shipping orders**: customer fills name, email, phone, shipping address in checkout form
3. Shop page calls `POST /api/checkout/create-payment-link` → Worker server-re-prices cart, calculates shipping, calls Square Payment Links API to create a one-time payment link with the exact total
4. Worker stores a pending order in KV (`paymentStatus: 'pending'`, Square `payment_link.id` + `order_id`)
5. Worker reserves stock (stores `reservation:<orderId>` KV key with 24h TTL, does NOT yet decrement `product.stock`)
6. Worker returns the Square payment link URL to the browser
7. Browser redirects customer to Square-hosted payment page (Square handles card UI, Apple Pay, Google Pay, receipts — no PCI scope on site)
8. Square fires `payment.completed` webhook → Worker receives at `POST /api/webhooks/square`, verifies signature (`SQUARE_WEBHOOK_SIGNATURE_KEY`), marks order `paymentStatus: 'paid'`, decrements real stock, releases reservation key, sends MailChannels email to Cynthia with order summary
9. Square redirects customer to `success_url` → `shop/order-success.html?orderId=<internalId>`
10. For **pickup orders**: still use existing `/api/orders` pending-invoice flow (no online payment, Square invoice sent manually)

**Why Payment Links over Checkout API:**
- No Square SDK or OAuth needed — just a REST call with the secret key
- Link is a hosted URL Cynthia can also share manually (Instagram, DMs)
- No PCI exposure on the site
- Square handles tax collection if configured in her account

### §38B addendum — H4 separate-inventory decision (resolved)

**Decision:** Each product belongs to **one** location — `hartland` or `fenton`. Stock counts are independent. If a piece exists at both shops, create two product entries (one per location). No "shared" or "both" mode.

**Rationale:** Cynthia's mental model: each shelf at each shop is its own thing. Shared-stock semantics (single SKU, two stock counts) would either confuse the count or require shop-specific decrements on every sale — more UI surface than benefit for the early stage. Two entries keeps every screen (Inventory, Sold-modal, Orders) trivially correct.

**Schema:**
- `product.location: 'fenton' | 'hartland'` (defaults to `'hartland'` when absent — all pre-existing pieces).
- Worker pass-through preserves the field on `PUT /api/products` without code change.
- Sale-source location is prepended to the order note as `[Hartland]` / `[Fenton]` (no worker-side schema change needed for May 19).

**UI surface added:**
- Product edit modal: `Location` select (required).
- Inventory toolbar: `filter-location` dropdown (Both / Hartland / Fenton).
- Inventory rows: small `HRT` / `FEN` badge next to product name.
- Ring-up-sale modal: `Where` select (remembers last choice in `localStorage.mr_last_location`).

**Future (if/when promoted to first-class):**
- Worker `/api/orders/in-store` should accept and persist `saleLocation` on the order envelope (currently lives only in the note).
- Dashboard split by location (today's close-out per shop).
